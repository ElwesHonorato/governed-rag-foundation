# Ticket: Distributed Chunking Pipeline (Spark + LangChain + Delta)

## Type
Feature

## Priority
High

## Context
The current RAG pipeline follows the stages:

`parse -> chunk -> embed`

The chunk stage must scale to large documents and large volumes of parsed artifacts stored in object storage.

To achieve enterprise-grade scalability and consistency with the rest of the architecture, chunking should be implemented using:

- Spark for distributed compute
- LangChain Text Splitters for chunking logic
- Delta Lake for chunk storage

This ensures that:

- chunking can scale horizontally
- chunk boundaries remain semantically meaningful
- chunks are stored in an optimized analytical format
- downstream embedding workers can process chunks efficiently

This ticket introduces a Spark-based chunk processing job integrated with the existing worker pipeline.

## Objective
Implement a Spark-powered chunking stage that:

1. Reads parsed document artifacts (JSON) from object storage.
2. Uses Spark for distributed processing.
3. Uses `LangChain RecursiveCharacterTextSplitter` for chunking logic.
4. Writes chunks to Delta Lake tables in object storage.
5. Produces deterministic chunk identifiers for downstream embedding and lineage.

## Architecture
```text
Queue Message
  |
  v
Chunk Worker
  |
  v
Spark Job
  |
  +-- Read JSON artifact
  |
  +-- Distributed chunking (LangChain splitter)
  |
  v
Chunks DataFrame
  |
  v
Delta Table (object storage)
  |
  v
Embed Worker (downstream)
```

## Input
Queue message contains:

- `document_id`
- `processed_document_key`
- `source_uri`
- `run_id`

The parsed artifact stored in object storage contains:

- `document_id`
- `text`
- `metadata`
- `content_type`

## Output
Chunks must be written to Delta Lake with the following schema.

| column | type | description |
|---|---|---|
| chunk_id | STRING | deterministic chunk identifier |
| document_id | STRING | parent document |
| chunk_index | INT | position in document |
| chunk_text | STRING | chunk content |
| source_uri | STRING | original file |
| processed_document_key | STRING | parsed artifact |
| content_type | STRING | mime type |
| created_at | TIMESTAMP | processing time |
| run_id | STRING | pipeline execution id |
| metadata | MAP<STRING, STRING> or JSON | additional metadata |

## Chunk ID
Chunk identifiers must be deterministic:

`chunk_id = sha256(document_id + ":" + chunk_index + ":" + sha256(chunk_text))`

This guarantees stable IDs across reprocessing runs.

## Storage
Chunks will be written to:

`s3://<bucket>/delta/chunks/`

Using Delta Lake format.

Recommended partitioning strategy:

`partitionBy(document_id)`

This allows efficient retrieval of document chunks.

## Implementation
### 1. Spark Chunk Job
Create a job module responsible for chunking logic.

Suggested location:

`pipeline_common/processing/spark_chunk_job.py`

Responsibilities:

- accept `SparkSession`
- receive input DataFrame
- perform distributed chunking
- return chunks DataFrame

### 2. Read Parsed Artifacts
Spark should read JSON artifacts directly from object storage.

Example:

```python
spark.read.json(object_path)
```

Support both:

- JSON lines
- multiline JSON

### 3. Flatten Document Structure
If artifacts contain nested sections or pages (for example `pages[]` or `sections[]`), flatten before chunking into normalized units such as:

- `document_id`
- `unit_index`
- `unit_text`

### 4. Chunking Logic
Use LangChain `RecursiveCharacterTextSplitter` with configuration keys:

- `chunk_size`
- `chunk_overlap`

Example behavior:

- `chunk_size = 500`
- `chunk_overlap = 50`

### 5. Distributed Execution
Chunking must run inside Spark executors.

Preferred approach: `mapInPandas`.

Reasons:

- avoids Python UDF overhead
- better performance for text workloads
- allows direct LangChain integration

The function should:

- receive batches of documents
- apply LangChain splitter
- emit chunk rows

### 6. Write Delta Output
Write the resulting chunks DataFrame:

`chunks_df.write.format("delta")`

Mode: `append` (configurable for development environments).

### 7. Worker Integration
Update the existing chunk worker service to:

- consume queue message
- resolve artifact path
- create Spark session
- run chunk Spark job
- write Delta output
- acknowledge message

Worker should remain thin orchestration only.

## Configuration
Extend the chunk processing config contract with:

- `chunk_size`
- `chunk_overlap`
- `delta_output_path`
- `write_mode`
- `multiline_json`

These settings should be injected through the existing configuration system.

## Testing
### Unit Tests
Verify:

- deterministic chunk IDs
- correct overlap behavior
- empty documents produce zero chunks

### Integration Test
Run Spark locally:

- load sample JSON
- execute chunk job
- confirm Delta table written
- validate schema and row counts

## Acceptance Criteria
The feature is considered complete when:

- chunk worker can process a parsed JSON artifact
- Spark performs distributed chunking
- LangChain splitter is used for chunk logic
- chunks are written to Delta Lake
- deterministic chunk IDs are produced
- embed worker can consume resulting chunks
